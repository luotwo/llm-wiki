原创 SinKris *2026年3月30日 18:07*

说一件让我有点尴尬的事。

去年年底做季度竞品更新，我花了整整两天。

第一天：挨个打开竞品官网、App Store 页面、微信小程序，截图，粘贴到飞书文档，手动整理产品功能、定价、最近的更新动态。

第二天：把五个竞品的信息并排放，开始写对比分析。写到一半发现某个竞品的定价页面昨天刚更新了，刚整理的数据作废，重新来。最后交出去的报告，产品 VP 看完说了一句话：

"格式和上季度不一样，能不能统一一下？"

我沉默了一下，打开上季度的报告对照着改。

又花了半天。

那次之后我就在想，竞品分析这件事，真正需要我来做的部分，其实只有一个： **判断这些竞品信息对我们意味着什么** 。信息收集、格式整理、横向对比，这些环节压根不需要人脑。

后来我用 Claude Code 配了一套竞品分析的工作流，这一次季度更新用了不到 2 小时。这篇文章把整个做法拆出来，你可以直接复用。

---

## 竞品分析为什么总是又慢又费力

在讲做法之前，先说清楚问题出在哪。

做竞品分析慢，不是因为竞品多，而是因为 **每个环节都有不必要的人工介入** ：

**信息收集靠手动。** 打开网页，复制粘贴，截图，整理到文档。每个竞品重复一遍，重复 N 遍。

**格式每次都不统一。** 这次用表格，上次用列表，新来的同事又整理成另一种样式。等到要横向比较的时候，发现根本没办法直接对比。

**数据容易过期。** 你今天整理的定价，下周竞品可能就更新了。但你不知道，继续用旧数据做判断。

**分析结论靠感觉。** 信息堆在那，"洞察"这一部分往往写的是"竞品 A 功能较为完善，建议我们关注"这类放之四海而皆准的废话。

这四个问题，在我配好工作流之后，基本全部解决了。

## 用到的工具：两个 Skill 的组合

这套流程用到两个东西的配合：

**`agent-browser`** （负责实时抓取）：让 Claude Code 直接打开竞品官网、定价页、App Store 详情页，提取最新信息。不是让 AI 去猜竞品情况，而是让它直接去看，保证数据是当天的。

**`competitive-profile-builder`** （负责结构化分析）：安装之后，Claude Code 会按照固定的竞品分析框架来整理和输出，每次格式完全一致，可以直接横向比较。

安装方式：

```
123456
# 安装 agent-browser（实时抓取能力）
npx skills add vercel-labs/agent-browser@agent-browser -g -y
 
# 安装 PM 竞品分析 Skill
claude plugin marketplace add phuryn/pm-skills
claude plugin install pm-market-research@pm-skills
```

两个都装上，下面的流程就可以跑了。

## 完整流程：分三步走

### 第一步：建立你的竞品清单文件（5 分钟）

新建一个文件夹，比如 `competitive/` ，在里面创建一个 `competitors.md` 文件，把你要分析的竞品列出来，格式如下：

```
123456789101112131415
# 竞品清单
 
## 竞品 A
- 官网：https://www.xxx.com
- 定价页：https://www.xxx.com/pricing
- App Store：https://apps.apple.com/...
- 重点关注：会员体系、定价策略

## 竞品 B
- 官网：https://www.yyy.com
- 定价页：https://www.yyy.com/pricing
- 重点关注：核心功能、最近更新

## 竞品 C
...
```

每个竞品给它需要访问的页面链接，以及这次分析你重点关注的维度。这个文件写完之后可以长期维护，下次季度更新直接复用，只需要改一下"重点关注"的方向。

---

### 第二步：让 Claude Code 实时抓取竞品信息（约 30-40 分钟）

在 Claude Code 里输入：

```
1234567891011
请读取 @competitive/competitors.md 里的竞品清单。
 
对每一个竞品，使用浏览器访问对应的链接，提取以下信息：
1. 核心产品定位（官网首页的主要价值主张）
2. 主要功能模块（产品页面列出的功能分类）
3. 定价体系（各套餐名称、价格、核心权益）
4. 近期更新（App Store 的版本更新记录，最近 3 个版本）
5. 用户评价亮点（App Store 评论里高频出现的正面和负面关键词）
 
每个竞品单独输出一个文件，保存到 competitive/profiles/ 文件夹，
文件名用竞品名命名，比如 competitive/profiles/competitor_a.md
```

这一步，Claude Code 会真正打开每一个链接，读取页面内容，提取你需要的信息。不是靠 AI 的训练数据去猜，是实时去看。

五个竞品大概需要 30 到 40 分钟，每个竞品生成一份完整的信息档案，存在 `profiles/` 文件夹里。

> “
> 
> **这里有一个重要的好处，值得单独说：**  
> 过去你用 ChatGPT 或者 Deep Research 去查竞品定价，得到的答案经常是几个月前的旧数据，因为 AI 的训练数据有截止日期。但 `agent-browser` 是真正打开浏览器去访问页面，拿到的是今天的价格，今天的功能，今天的版本更新。
> 
> 这个差别，在竞品定价和产品功能变动频繁的行业里，非常关键。

---

### 第三步：生成结构化对比报告（约 20 分钟）

竞品档案都有了，最后一步是横向对比、输出结论。

在 Claude Code 里继续输入：

```
12345678910111213141516171819
请读取 @competitive/profiles/ 里的所有竞品档案，完成以下分析：
 
1. 功能对比矩阵
   以表格形式呈现各竞品的核心功能覆盖情况，用 ✓ / ✗ / △ 标注

2. 定价对比
   各竞品的定价区间、套餐结构、性价比定位

3. 差异化特征
   每个竞品最突出的 1-2 个差异化方向

4. 市场空白
   综合对比之后，目前竞品都没有做好或没有覆盖的方向

5. 对我们的启示
   基于以上分析，值得我们关注或借鉴的 3 条具体建议

输出为 competitive/comparison_report.md
同时输出一个适合在评审会上直接使用的一页纸摘要：competitive/executive_summary.md
```

20 分钟后，你会得到两份文件：

一份是完整的对比报告，有功能矩阵、定价对比、差异化分析；另一份是一页纸摘要，可以直接带去产品评审或者向上汇报，不需要再另外整理。

## 有几件事值得提前知道

**关于抓取成功率：** 有些网站有反爬机制，或者需要登录才能看到完整定价信息，这种情况下 `agent-browser` 可能拿不到全部数据。遇到这类情况，对应的竞品手动补充就好，其他的照常跑。

**关于竞品档案的维护：** 建议每次更新不要覆盖旧档案，而是新建一个带日期的版本，比如 `competitor_a_2026Q1.md` 。这样随着时间积累，你会有一份竞品的变化历史，对判断竞品的战略走向很有帮助。

**关于分析结论的使用：** 第三步输出的"对我们的启示"是 AI 基于信息对比给出的建议，不是替代你的战略判断。你需要结合自己对用户、市场、团队资源的理解，来决定哪条建议值得真正落地。工具做信息层，判断层还是你的。

## 完整操作流程一览

```
123456789101112131415161718
【准备】建立 competitive/ 文件夹
  → 创建 competitors.md，列出竞品名称和链接
  → 耗时：5 分钟

【Step 1】实时抓取竞品信息
  → 指令：访问每个竞品链接，提取定位 / 功能 / 定价 / 更新 / 评价
  → 产出：competitive/profiles/ 里的 N 份竞品档案
  → 耗时：约 30-40 分钟（5个竞品）

【Step 2】生成对比报告
  → 指令：横向对比，输出功能矩阵 + 定价对比 + 差异化 + 启示
  → 产出：comparison_report.md + executive_summary.md
  → 耗时：约 20 分钟

【你来做】
  → 读报告，验证关键数据是否准确
  → 结合业务判断，筛选真正值得行动的结论
  → 带 executive_summary 进评审
```

**从启动到拿到可用报告：不超过 2 小时。**

## 最后说一句

做竞品分析这件事，PM 和运营真正需要输出的，是 **判断** ：这个市场现在的格局是什么，竞品在往哪个方向走，我们的机会在哪里。

但大多数人把大半时间花在了判断之前的环节——收集、整理、对齐格式。

这套流程要改变的，就是这个比例。

让信息收集和格式整理回归它应有的成本：低，快，可以随时重跑。然后把省出来的时间，全部放在那个真正需要你来做的部分上。
